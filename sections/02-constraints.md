# 02. Constraints

> Anything that constrains teams in design and implementation decisions or decision about related processes. Can sometimes go beyond your own system boundaries.

---

## 2.1 Technical Constraints

> It can be language, framework, platform, tool, or any other technical constraint that needs to be followed.

| Constraint       | Background / Motivation                                                    |
| ---------------- | -------------------------------------------------------------------------- |
| e.g., TypeScript | Strongly typed language to improve code quality and reduce runtime errors. |
| e.g., PostgreSQL | Required by company standards for relational data storage.                 |

## 2.2 Organizational Constraints

> It can be organizational policy, process, or any other organizational constraint that needs to be followed.

| Constraint        | Background / Motivation                                                      |
| ----------------- | ---------------------------------------------------------------------------- |
| e.g., Team limits | Team consists of only 4 developers, which limits the development velocity.   |
| e.g., Budget      | Maximum compute budget is capped at $500/month for the initial launch phase. |

## 2.3 Conventions

> It can be coding standard, design pattern, or any other convention that needs to be followed.

| Convention                | Background / Motivation                                                            |
| ------------------------- | ---------------------------------------------------------------------------------- |
| e.g., RESTful APIs        | Standard integration protocol used across all of the company's internal services.  |
| e.g., Semantic Versioning | Ensures predictable version numbers for all software releases (Major.Minor.Patch). |
